Why doesn't your flow data match the ATMS data?
The ATMS system and the PeMS system use slightly different definitions for their time periods which cause the flows to be slightly off. In order to compute a 5-minute aggregate from the 30-second flows, the ATMS system uses data in the 5-minutes leading up to the time period but the PeMS system uses the 5-minutes after the time period. For example, to compute the aggregate flow for the time period of 5:15:00, the ATMS system uses the 30-second data in the range of 5:10:01 to 5:14:59 and the PeMS system uses the rage of 5:15:00 until 5:19:59. In short, we time stamp at the beginning of the time period from which the data is collected.

There is a debate as to whether it's better to time stamp at the beginning or at the end of the time period. We believe that time stamping values at the end of the time period has the potential to confuse people if applied consistently to all time scales. For example, we aggregate the data from 30-seconds up to many different time scales. If we applied the same scheme for the 5-minute data to the 1-hour data - that the time stamp for the aggregated data is set at the end of the aggregation period - then the data from 1pm until 2pm would have a time stamp of 2pm. At the daily aggregation level, all of the data for 3/14/03 00:00:00 until 3/14/03 23:59:59 would be stamped at 3/15/03. All of the data in the month of March would be time stamped with April, and so on. This seems confusing. If we make the obvious correction, and time stamp everything above the 5-minute data with the start of the time period, then we're switching back and forth between the two time stamping schemes. To some people this might be confusing as well. So we chose the scheme that could be applied consistently at all time scales with what we thought would be the least amount of confusion.

Note that the speeds will never match due to the fact that the PeMS system uses a different calculation in order to compute speed.

How do you deal with sensors that measure speed directly?
A majority of the loop detectors in the State of California are single loop detectors. This means that there is only a single detector station in a lane. They can't measure speed directly and it has to be computed (an explanation of how we do this can be found in the create_anchor(null, 'Help', 'help_calc') ?> page). But there are a couple of sensors on the freeway which measure speed directly. These are either double loop detectors or side-fire radar. For the double loop detectors, our research has shown that our speed calculation algorithms based solely on the flow and occupancy are better than the speed measurements from the double loops. Therefore we don't use the double loop measured speeds in our algorithms. We have not had a chance to investigate the fidelity of the speed measurements coming from the side-fire radar. Hence we are currently using our own speed algorithm.

import pandas as pd
import os

# Set the path where your .xlsx files are stored
data_folder = 'path_to_your_files'

# Define the station metadata manually (you already have it)
station_metadata = {
    715898: {"Type": "Mainline", "Lat": 33.880367, "Lon": -118.021489},
    715901: {"Type": "OnRamp", "Lat": 33.883681, "Lon": -118.027733},
    # etc... for each station
}

# Placeholder list to collect per-station DataFrames
all_stations_data = []

# Loop through your station IDs
for station_id, meta in station_metadata.items():
    # Build expected filenames based on your naming scheme
    if meta['Type'] == 'Mainline':
        flow_occ_file = f'Mainline VDS {station_id} Flow & Occup.xlsx'
        speed_q_file = f'Mainline VDS {station_id} Speed & Q.xlsx'
    else:  # OnRamp or OffRamp
        flow_occ_file = f'{meta["Type"]} VDS {station_id} Flow & Occup.xlsx'
        speed_q_file = None  # No speed/Q for ramps

    # Read the flow/occupancy data
    flow_occ_path = os.path.join(data_folder, flow_occ_file)
    flow_occ_df = pd.read_excel(flow_occ_path)

    # If Mainline, also read speed/Q
    if speed_q_file:
        speed_q_path = os.path.join(data_folder, speed_q_file)
        speed_q_df = pd.read_excel(speed_q_path)
        # Merge flow_occ_df and speed_q_df on the time column
        merged_df = pd.merge(flow_occ_df, speed_q_df, on='5 Minutes')
    else:
        merged_df = flow_occ_df.copy()

    # Add metadata columns
    merged_df['Station_ID'] = station_id
    merged_df['Station_Type'] = meta['Type']
    merged_df['Latitude'] = meta['Lat']
    merged_df['Longitude'] = meta['Lon']

    # Append to the full list
    all_stations_data.append(merged_df)

# Concatenate all station DataFrames into one big DataFrame
full_data = pd.concat(all_stations_data, ignore_index=True)

# Save master DataFrame
full_data.to_pickle('full_station_data.pkl')  # Fast loading
full_data.to_csv('full_station_data.csv', index=False)  # Human-readable backup

# Choose one station as your reference point (e.g. first/mainline station)
ref_lat = station_df.iloc[0]['Latitude']
ref_lon = station_df.iloc[0]['Longitude']

def haversine_dist(lat1, lon1, lat2, lon2):
    # Approximate distance on Earth in km
    R = 6371
    lat1, lon1, lat2, lon2 = map(np.radians, [lat1, lon1, lat2, lon2])
    dlat = lat2 - lat1
    dlon = lon2 - lon1
    a = np.sin(dlat/2)**2 + np.cos(lat1)*np.cos(lat2)*np.sin(dlon/2)**2
    return R * 2 * np.arcsin(np.sqrt(a))

station_df['dist_from_start_km'] = station_df.apply(
    lambda row: haversine_dist(ref_lat, ref_lon, row['Latitude'], row['Longitude']),
    axis=1
)

station_df.sort_values(by='dist_from_start_km', inplace=True)

Exactly—think of exit_zone (or merge_zone for on-ramps) as a simple flag that tells your model “treat flow at this station differently.” It’s just a 0/1 column, where:

0 means “this is a normal stretch of freeway with very consistent upstream→downstream flow”
